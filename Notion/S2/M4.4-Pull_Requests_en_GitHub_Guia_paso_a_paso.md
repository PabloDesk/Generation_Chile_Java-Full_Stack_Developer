# **Pull Requests** en GitHub- Guía paso a paso

# 🌟 Guía paso a paso para **Colaborar** 🌟

> Objetivo: Dejar en Notion un instructivo ultra detallado para trabajar colaborativamente con Pull Requests (PR), cubriendo dos flujos:
> 
> 1. **Fork → PR al repo principal**
> 2. **Colaborador directo** (branch en el repo principal → PR interno)
> 
> Incluye: pasos de interfaz y consola, buenas prácticas, plantillas, cómo sincronizar cambios, y resolución de conflictos.
> 

---

## Conceptos clave (rápido)

- **Fork:** tu copia del repo principal en tu cuenta. Ideal para colaborar sin permisos directos.
- **Remotos:**
    - `origin` → tu Fork (o el repo principal si eres colaborador).
    - `upstream` → el repo principal (solo en flujo por Fork).
- **PR:** solicitud de integrar tus cambios a otra rama (normalmente `main` del repo principal).
- **Ramas (branches):** trabajas **siempre** en rama distinta a `main`.

> 💡 Regla #1: Nunca trabajes directo en main. Crea una rama por cada feature/fix.
> 

---

## Checklist previo (ambos flujos)

- [ ]  Tener Git instalado y configurado (`git --version`, `git config user.name`, `git config user.email`).
- [ ]  Decidir el **nombre de la rama** (corto, descriptivo, sin espacios): `feature/login-form`, `fix/typo-readme`.
- [ ]  Revisar si hay **reglas de PR** (plantillas, CI, tests obligatorios).
- [ ]  Acordar **convenciones de commits** (p. ej., Conventional Commits).
- [ ]  Confirmar la **rama base** para PR (normalmente `main`).

## Flujo A — **Fork → PR al repo principal**

---

### A1. Hacer **Fork**

1. Ve al repo principal en GitHub.
2. Haz clic en **Fork** (arriba a la derecha) → crea el fork en tu cuenta.
3. ✅ Resultado: tienes `tu-usuario/repo` (copia del original).

> ⚠️ Posible error: “¿Por qué no veo mis cambios en el repo principal?”
> 
> 
> Porque estás en **tu Fork**. Los cambios llegan al repo principal **solo** al crear una **PR** desde tu Fork.
> 

---

### A2. Clonar tu Fork

```bash
# Clona tu fork (NO el repo principal)
git clone https://github.com/TU_USUARIO/REPO.git
cd REPO
git remote -v  # Debe mostrar origin -> TU fork

```

---

### A3. Configurar `upstream` (repo principal)

```bash
# Agrega el remoto del repo principal (lee su URL desde GitHub)
git remote add upstream https://github.com/ORG_O_AUTOR/REPO.git
git remote -v  # Debe mostrar origin (tu fork) y upstream (repo principal)

```

> 🧭 ¿Por qué es importante? Podrás sincronizar tu Fork con los cambios del repo principal.
> 

---

### A4. Crear rama de trabajo

```bash
git fetch upstream
git checkout main
git pull upstream main   # Trae lo último del repo principal a tu local
git push origin main     # Opcional: alinea tu fork en GitHub

# Crea tu rama de trabajo desde main actualizado
git checkout -b feature/nombre-corto

```

> ⚠️ Error común: crear la rama sin actualizar main. Luego tendrás conflictos.
> 
> 
> ✅ **Siempre**: `fetch upstream` + `pull upstream main` antes de crear la rama.
> 

---

### A5. Commits y push a tu Fork

1. Trabaja y **guarda cambios**:
    
    ```bash
    git status
    git add .
    git commit -m "feat: agrega formulario de login con validación básica"
    
    ```
    
2. Sube la rama a tu Fork:
    
    ```bash
    git push -u origin feature/nombre-corto
    
    ```
    
    El `-u` vincula tu rama local con la remota en `origin`.
    

> 💡 Pequeños commits y mensajes claros facilitan la revisión.
> 

---

### A6. Crear la Pull Request hacia el repo principal

1. En GitHub (tu Fork), verás un banner: **“Compare & pull request”** → clic.
2. Verifica:
    - **base repository:** `ORG_O_AUTOR/REPO`
    - **base branch:** `main` (o la que indique el proyecto)
    - **head repository:** `TU_USUARIO/REPO`
    - **compare:** `feature/nombre-corto`
3. Completa **título** y **descripción** (usa la plantilla del proyecto si existe).
4. Marca **checklist** (si hay), adjunta capturas si aplica.
5. Crea la PR.

> ⚠️ Errores posibles:
> 
> - Base/compare invertidos → harías PR a tu propio Fork. **Revisa los selectores**.
> - CI falla por tests → revisa logs, corrige y vuelve a pushear a la **misma rama** (la PR se actualiza sola).

---

### A7. Revisión y merge en el repo principal

- Los mantenedores revisan: comentarios, “**Changes requested**” o “**Approve**”.
- Si piden cambios:
    
    ```bash
    # haz cambios, luego:
    git add .
    git commit -m "fix: corrige validación de input vacío"
    git push
    # (La PR se actualiza automáticamente)
    
    ```
    
- Cuando aprueben, harán **Merge** (normal, squash o rebase, según política).

> 💡 Tip: Responde comentarios punto por punto; muestra qué cambiaste.
> 

---

### A8. **Sincronizar** tu Fork y tu local

> Después del merge, todos deben actualizar su local y su fork.
> 

**En tu equipo/local:**

```bash
git checkout main
git fetch upstream
git pull upstream main   # trae los últimos cambios del repositorio principal
git push origin main     # actualiza tu fork en GitHub

```

**(Opcional) limpiar ramas ya fusionadas:**

```bash
git branch -d feature/nombre-corto                 # local
git push origin --delete feature/nombre-corto      # remoto en tu fork

```

> ⚠️ Error común: seguir trabajando sin hacer pull upstream main.
> 
> 
> Resultado: trabajar sobre una base vieja → más conflictos luego.
>

## Flujo B — **Colaborador directo** (rama dentro del repo principal)

> Úsalo cuando tienes permisos de escritura en el repo principal.
> 

---

### B1. Crear rama desde `main`

```bash
git clone https://github.com/ORG_O_AUTOR/REPO.git
cd REPO
git checkout main
git pull origin main
git checkout -b feature/nombre-corto

```

---

### B2. Commits, push y PR interna

```bash
git add .
git commit -m "feat: agrega componente Header con responsive"
git push -u origin feature/nombre-corto

```

En GitHub del **repo principal** verás: **“Compare & pull request”** → clic, completa descripción y crea la PR.

> 🔐 Si main está protegida, no podrás pushear directo (¡bien!). Debes crear PR sí o sí.
> 

---

### B3. Revisión, merge y **pull** para sincronizar

- Se revisa la PR y se hace **Merge**.
- **Todos** actualizan su `main` local:
    
    ```bash
    git checkout main
    git pull origin main
    
    ```
    
- Limpieza de rama:
    
    ```bash
    git branch -d feature/nombre-corto
    git push origin --delete feature/nombre-corto
    
    ```

## Plantillas útiles (PR y commits)

### 📝 Plantilla de Pull Request (copiar/pegar)

```markdown
## Descripción
Breve explicación del cambio: qué problema resuelve y cómo.

## Tipo de cambio
- [ ] Fix (corrección de bug)
- [ ] Feature (nueva funcionalidad)
- [ ] Docs (documentación)
- [ ] Refactor / Perf / Chore

## ¿Cómo probar?
1. Pasos para reproducir el entorno
2. Comandos (si aplica)
3. Capturas o evidencia (si aplica)

## Checklist
- [ ] Probado localmente
- [ ] Sin romper compatibilidad
- [ ] Agregué/actualicé documentación
- [ ] Aprobé linters/tests (si aplica)

```

### ✍️ Ejemplos de **Conventional Commits**

- `feat: agrega validación de email en el registro`
- `fix: corrige división por cero en calculadora`
- `docs: actualiza guía de contribución`
- `refactor: extrae lógica de parseo a util`
- `test: agrega casos límite para raíz n-ésima`

## Buenas prácticas y reglas de oro

- **1 rama = 1 objetivo**. Evita PR gigantes.
- **Actualiza antes de empezar** (`pull upstream main` o `pull origin main`).
- **Commits atómicos** y mensajes claros.
- **No** mezcles formateos masivos con cambios funcionales.
- Resuelve **conflictos** localmente y **explica** si tocaste archivos de otros.
- Después del **merge**, **todos** deben hacer `git pull` (o `fetch` + `merge/rebase`).

---

## Errores comunes & cómo resolverlos

### “Hice PR, pero apunta a mi Fork”

- Te equivocaste en **base/head** al crearla.
- **Solución:** cerrar esa PR y crear una nueva asegurando:
    - **base repo:** `ORG/REPO`
    - **base branch:** `main`
    - **head repo:** `TU_USUARIO/REPO` (fork)
    - **compare:** `feature/...`

### “Conflictos al actualizar”

```bash
git fetch upstream
git checkout feature/nombre-corto
git merge upstream/main   # o: git rebase upstream/main
# Edita archivos en conflicto, luego:
git add .
git commit               # (si estabas en merge)
# o si rebase:
git rebase --continue
git push                 # si rebase: git push --force-with-lease

```

> ⚠️ Usa `--force-with-lease` (no `--force`) para no pisar trabajo ajeno.
> 

### “No me deja pushear a main”

- **Rama protegida** (correcto). Crea **PR** desde una rama.

### “Mi Fork está desactualizado”

```bash
git checkout main
git fetch upstream
git pull upstream main
git push origin main

```

## FAQ express

**Q: ¿Puedo trabajar sin Fork si tengo permisos?**

A: Sí, usa el **Flujo B** con ramas internas y PR interna.

**Q: ¿Merge, squash o rebase al integrar PR?**

A: Depende de la política del repo. *Squash* deja un historial más limpio por PR.

**Q: ¿Cada cuánto sincronizo mi rama con `main`?**

A: En equipos activos, **diariamente** o antes de finalizar la tarea.
