# Cómo crear una tabla en SQL

## 🎯 Objetivo de esta página

Comprender **cómo se construye una base de datos desde cero**, creando tablas paso a paso, insertando datos, consultándolos y entendiendo **por qué SQL funciona así**.

---

## 🧩 ¿Qué representa una tabla?

En una base de datos relacional:

- Una **tabla representa una entidad**
- Una **entidad** es algo del mundo real que queremos modelar

Ejemplos de entidades:

- Cliente
- Producto
- Proveedor
- Compra

Cada tabla:

- Tiene **filas** → registros
- Tiene **columnas** → atributos

---

## 🧍 Paso 1 – Entidad Cliente

Un **cliente** es una entidad común en muchos sistemas.

Información básica de un cliente:

- Nombre
- RUT
- Email

---

## 🏗️ Crear tabla `clientes`

```sql
CREATE TABLE clientes (
    id_cliente INT PRIMARY KEY,
    nombre VARCHAR(100),
    rut VARCHAR(15),
    email VARCHAR(100)
);

```

### Qué representa esta tabla

- `clientes` → entidad
- `id_cliente` → identificador único (PK)
- Cada fila → un cliente distinto

---

## 🧪 Insertar datos en clientes

```sql
INSERT INTO clientes VALUES
(1, 'Ana Pérez', '12.345.678-9', 'ana@mail.com'),
(2, 'Juan Soto', '9.876.543-2', 'juan@mail.com');
```

---

## 🔍 Consultar clientes

```sql
SELECT * FROM clientes;
```

Aquí se puede ver:

- La estructura real de la tabla
- Los datos almacenados
- Cómo SQL devuelve filas y columnas

---

## 📦 Paso 2 – Entidad Producto

Los clientes compran **productos**, por lo tanto los productos deben existir como **otra entidad**.

---

## 🏗️ Crear tabla `productos`

```sql
CREATE TABLE productos (
    id_producto INT PRIMARY KEY,
    nombre VARCHAR(100),
    precio INT
);
```

---

## 🧪 Insertar datos en productos

```sql
INSERT INTO productos VALUES
(1, 'Notebook', 800000),
(2, 'Mouse', 15000),
(3, 'Teclado', 30000);
```

---

## 🔍 Consultar productos

```sql
SELECT * FROM productos;
```

Hasta ahora:

- Cada entidad vive en su propia tabla
- No hay datos mezclados
- Cada tabla tiene una responsabilidad clara

---

## 🧾 Paso 3 – Relación Cliente ↔ Producto

Reglas del negocio:

- Un cliente puede comprar **uno o muchos productos**
- Un producto puede ser comprado por **muchos clientes**

Este tipo de relación es:

> Muchos a muchos
> 

Para representarla, se crea una **tabla intermedia**.

---

## 🏗️ Crear tabla `compras`

```sql
CREATE TABLE compras (
    id_compra INT PRIMARY KEY,
    id_cliente INT,
    id_producto INT,
    fecha DATE,
    cantidad INT
);
```

Esta tabla representa:

- Qué cliente compró
- Qué producto
- Cuándo
- Cuántos

---

## 🧪 Insertar datos en compras

```sql
INSERT INTO compras VALUES
(1, 1, 1, '2024-09-01', 1),
(2, 1, 2, '2024-09-05', 2),
(3, 2, 3, '2024-09-10', 1);
```

---

## 🔍 Consultar compras (sin JOIN)

```sql
SELECT * FROM compras;
```

```sql
SELECT * FROM compras
WHERE id_cliente = 1;
```

### Qué se observa aquí

- Se ven **IDs**, no nombres
- Las relaciones existen mediante identificadores
- Cada tabla guarda **solo su información**

Esto es **correcto y deseado**.

---

## 🧠 ¿Por qué no veo el nombre del cliente o del producto?

Porque:

- La tabla `compras` **no guarda nombres**
- Solo guarda referencias (`id_cliente`, `id_producto`)
- Los datos reales están en otras tablas

Esto evita:

- Duplicar información
- Inconsistencias
- Problemas de mantenimiento

---

## 🏭 Paso 4 – Entidad Proveedor

Los productos pueden tener **proveedores**.

Un proveedor es otra entidad independiente.

---

## 🏗️ Crear tabla `proveedores`

```sql
CREATE TABLE proveedores (
    id_proveedor INT PRIMARY KEY,
    nombre VARCHAR(100),
    contacto VARCHAR(100)
);
```

---

## 🧪 Insertar proveedores

```sql
INSERT INTO proveedores VALUES
(1, 'Proveedor A', 'contacto@proveedora.com'),
(2, 'Proveedor B', 'contacto@proveedorb.com');

```

---

## 🔍 Consultar proveedores

```sql
SELECT * FROM proveedores;

```

---

## 🔗 Relación Producto ↔ Proveedor

Reglas del negocio:

- Un producto puede tener **uno o muchos proveedores**
- Un proveedor puede vender **muchos productos**

Esto genera otra relación **muchos a muchos**.

---

## 🏗️ Crear tabla `producto_proveedor`

```sql
CREATE TABLE producto_proveedor (
    id INT PRIMARY KEY,
    id_producto INT,
    id_proveedor INT
);
```

---

## 🔍 Consultar relación producto–proveedor

```sql
SELECT * FROM producto_proveedor;
```

Aquí se repite el patrón:

- Entidades principales
- Tablas intermedias para relaciones

---

## 🧠 Hasta aquí, ¿cuál es el propósito de SQL?

SQL permite:

- Definir estructuras (tablas)
- Guardar información
- Consultar datos por partes
- Mantener datos organizados
- Preparar información para ser cruzada

Por ahora:

- Las consultas trabajan **una tabla a la vez**
- Las relaciones existen, aunque no se crucen aún

---

## 📍 Paso 5 – Atomizar información: Dirección

Una dirección puede guardarse de forma simple:

```sql
direccion VARCHAR(200)
```

Esto **funciona** y es válido.

---

## 🤔 ¿Cuándo separar la dirección?

Separar la información tiene sentido cuando:

- Se repite mucho
- Se necesita filtrar o agrupar
- Tiene valor propio

Ejemplo:

- Comuna
- Región

---

## 🧱 Ejemplo de dirección atomizada

```sql
CREATE TABLE regiones (
    id_region INT PRIMARY KEY,
    nombre VARCHAR(100)
);
```

```sql
CREATE TABLE comunas (
    id_comuna INT PRIMARY KEY,
    nombre VARCHAR(100),
    id_region INT
);
```

```sql
ALTER TABLE clientes
ADD id_comuna INT;
```

---

## ❌ ¿Cuándo no atomizar?

No suele ser necesario separar:

- Comentarios
- Observaciones
- Descripciones libres
- Textos que no se filtran ni analizan

---

## 🧠 Ideas clave para cerrar esta página

- Cada tabla representa una **entidad**
- Las relaciones se representan con **IDs**
- Ver solo IDs es una señal de buen diseño
- SQL organiza y consulta datos de forma estructurada
- Atomizar datos es una **decisión**, no una obligación
- Diseñar bases de datos es **pensar antes de escribir SQL**
