# Plataformas de Repositorios

Las plataformas de repositorios alojan proyectos Git para colaborar, revisar código y automatizar tareas. Las tres más comunes son **GitHub**, **GitLab** y **Bitbucket**. Aquí aprenderás qué ofrecen, cómo elegir y cómo dar tus primeros pasos.

---

### ¿Qué es una plataforma de repositorios?

Un servicio que:

- Guarda tu código y su historial de cambios usando Git
- Permite colaboración con Pull Requests/Merge Requests y revisiones
- Ofrece seguimiento de tareas con Issues y tableros
- Integra CI/CD para probar y desplegar automáticamente
- Administra permisos y roles del equipo

<aside>

> Consejo: Git es la base. Estas plataformas no reemplazan a Git, lo potencian con funciones de equipo y cloud.
> 
</aside>

---

### Conceptos clave para principiantes

- Repositorio remoto: copia del repo en la plataforma
- Fork: copia de un repo de otra persona para proponer cambios
- Pull Request (GitHub/Bitbucket) / Merge Request (GitLab): solicitud de integración de cambios
- Issue: ticket para reportar errores o planear tareas
- CI/CD: pipelines que ejecutan pruebas y despliegues
- Wiki y README: documentación del proyecto

---

### [GitHub](https://www.github.com/)

- Enfoque: Comunidad enorme y ecosistema muy amplio
- Destaca en: Pull Requests, GitHub Actions, Projects, Codespaces
- Ideal para: Portafolios, open source, empresas de todos los tamaños
- Primeros pasos:
    1. Crea cuenta y un repo nuevo en GitHub
    2. Conecta tu repo local: `git remote add origin <URL>`
    3. Sube tu rama principal: `git push -u origin main`
    4. Crea una rama de feature y abre un Pull Request

### [GitLab](https://about.gitlab.com/)

- Enfoque: Plataforma DevOps integral end-to-end
- Destaca en: CI/CD integrado muy potente y autoalojado
- Ideal para: Equipos que requieren control on‑prem o compliance
- Primeros pasos:
    1. Crea cuenta o instancia self‑hosted
    2. Crea un proyecto y agrega remoto `origin`
    3. Activa CI con `.gitlab-ci.yml` y revisa el pipeline

### [Bitbucket](https://bitbucket.org/)

- Enfoque: Integración nativa con el ecosistema Atlassian
- Destaca en: Pipelines, y vínculo con Jira/Confluence
- Ideal para: Equipos que ya usan Jira y quieren integración fluida
- Primeros pasos:
    1. Crea cuenta y un repo en Bitbucket
    2. Configura `origin` y Pipelines si necesitas CI

---

### ¿En qué se parecen y en qué se diferencian?

- En común
    - Repos Git, PR/MR, Issues, permisos, CI/CD, Wiki, API
- Diferencias rápidas
    - GitHub: mayor comunidad y marketplace de acciones
    - GitLab: suite DevOps completa y fácil de autoalojar
    - Bitbucket: mejor encaje con Jira/Confluence

| **Plataforma** | CI/CD | Autoalojado | Fuerte en |
| --- | --- | --- | --- |
| GitHub | Actions | Vía Enterprise Server | Comunidad y ecosistema |
| GitLab | CI integrado | Sí, Community/Enterprise | DevOps end‑to‑end |
| Bitbucket | Pipelines | Data Center | Integración con Jira |

---

### Cómo elegir si estás empezando

- Si quieres visibilidad y open source: GitHub
- Si necesitas CI/CD potente integrado y opción on‑prem: GitLab
- Si tu equipo usa Jira/Confluence: Bitbucket

> Tip: Para tu portafolio personal, GitHub suele ser la elección más directa.
> 

---

### Checklist de inicio rápido

1. Crea una cuenta en la plataforma elegida
2. Crea un repositorio vacío con README y `.gitignore`
3. Conecta tu repo local con `git remote add origin <URL>`
4. Sube tu primera versión con `git push -u origin main`
5. Crea una rama, haz cambios y abre tu primer PR/MR
6. Activa CI/CD básico y agrega una tarea en Issues

---

### Buenas prácticas para nuevos estudiantes

- Un commit pequeño y descriptivo por cambio
- Ramas por funcionalidad, no trabajes siempre en `main`
- Abre PR/MR temprano para recibir feedback
- Usa Issues para planificar y documenta en README
- Activa CI para ejecutar pruebas automáticamente
- Protege `main` con revisiones y checks obligatorios

---

### Seguridad y permisos (lo esencial)

- Usa claves SSH o tokens personales para autenticarte
- Configura colaboradores con los mínimos permisos necesarios
- Activa escaneo de dependencias y revisa alertas
- Evita subir secretos. Usa variables de entorno o gestores de secretos

---

### Glosario rápido

- PR/MR: propuesta de integrar cambios de una rama
- Pipeline: secuencia automatizada de pasos de CI/CD
- Runner/Agent: ejecutor de los jobs del pipeline
- Release/Tag: versión marcada del código

---

### 💡 Otras alternativas (breve)

- **Azure DevOps**: Repos, Pipelines y Boards dentro del ecosistema Microsoft
- **SourceForge**: veterano del open source, aún activo
- **Gitea**: ligero y open source, ideal para autoalojado
- **AWS CodeCommit**: repos privados gestionados en AWS

---

### Recomendaciones

- Publica uno o dos proyectos simples en tu cuenta
- Prueba abrir PR/MR en un repo de práctica y solicita revisión
- Configura un pipeline que ejecute pruebas al hacer push
