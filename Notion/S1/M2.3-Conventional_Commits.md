# Conventional Commits

## 🧠 **Definición**

**Conventional Commits** es una **convención para estructurar mensajes de commit** siguiendo un formato semántico y legible, diseñado para mejorar la trazabilidad, automatización y consistencia del historial de cambios en un repositorio Git.

Se utiliza ampliamente en entornos profesionales y proyectos open source porque **estandariza la comunicación entre desarrolladores**, facilita la generación de changelogs, y se integra con herramientas de CI/CD y versionado semántico (*Semantic Versioning*).

---

## ⚙️ **Formato general**

```bash
<tipo>(<ámbito>): <mensaje breve>
```

### Componentes

| Elemento | Descripción |
| --- | --- |
| **tipo** | Indica la categoría del cambio realizado (feature, fix, docs, etc.) |
| **ámbito (scope)** | Opcional. Especifica el módulo o parte del sistema afectada. |
| **mensaje breve** | Descripción concisa del cambio en tiempo presente, sin punto final. |

---

## 🧩 **Tipos más comunes**

| Tipo | Descripción (ES) | Descripción (EN) |
| --- | --- | --- |
| **feat** | Nueva funcionalidad. | Introduces a new feature. |
| **fix** | Corrección de errores. | Fixes a bug. |
| **docs** | Cambios en documentación. | Documentation updates only. |
| **style** | Cambios de formato o estilo (espacios, comas, etc.) sin afectar la lógica. | Code style or formatting changes, no logic impact. |
| **refactor** | Reescritura o mejora del código sin cambiar su comportamiento. | Code refactoring without behavior change. |
| **perf** | Mejoras de rendimiento. | Performance improvements. |
| **test** | Agrega o modifica pruebas. | Adds or modifies tests. |
| **build** | Cambios en dependencias, herramientas o entorno de compilación. | Changes affecting the build system or dependencies. |
| **ci** | Cambios en configuración de integración continua. | Continuous integration changes. |
| **chore** | Tareas de mantenimiento o cambios menores. | Routine tasks or maintenance. |
| **revert** | Reversión de un commit anterior. | Reverts a previous commit. |

## 🧱 **Ejemplos de uso**

### 🔹 Español

```bash
feat(usuario): agregar validación de correo electrónico

fix(api): corregir error en el endpoint de autenticación

docs(readme): actualizar guía de instalación

refactor(servicio): simplificar la lógica de registro de usuarios

style(ui): ajustar márgenes en el componente de perfil

test(login): agregar pruebas unitarias para el flujo de inicio de sesión
```

### 🔹 English

```bash
feat(user): add email validation

fix(api): resolve authentication endpoint issue

docs(readme): update installation guide

refactor(service): simplify user registration logic

style(ui): adjust profile component margins

test(login): add unit tests for login flow
```

## 🚀 **¿Por qué usar Conventional Commits?**

| Beneficio | Descripción |
| --- | --- |
| **Estandarización** | Todos los commits siguen el mismo formato, facilitando la lectura. |
| **Automatización** | Permite generar changelogs y versiones automáticamente (ej. `semantic-release`). |
| **Trazabilidad** | Facilita entender qué cambió, dónde y por qué. |
| **Integración con CI/CD** | Los pipelines pueden interpretar tipos de cambios y ajustar versiones automáticamente (por ejemplo, `feat` incrementa la versión *minor*, `fix` incrementa la versión *patch*). |
| **Mejor colaboración** | Facilita code reviews y auditorías en equipos grandes. |

## 🔖 **Ejemplo completo**

```bash
feat(auth): implement JWT authentication

- Added login and signup endpoints
- Configured JWT token expiration
- Updated user model with token fields
```

```bash
fix(database): corregir error de conexión en entorno de producción

- Ajuste en variables de entorno
- Mejor manejo de errores de conexión
```

---

## 📜 **Resumen técnico**

- **Convención:** estándar de escritura para mensajes de commit.
- **Objetivo:** mejorar legibilidad, trazabilidad y automatización.
- **Formato:** `<tipo>(<ámbito>): <mensaje>`.
- **Compatible con:** Semantic Versioning, changelogs automáticos, CI/CD.
- **Buenas prácticas:**
    - Usar tiempo presente y evitar punto final.
    - Incluir un mensaje claro y conciso.
    - Mantener consistencia en todos los commits del equipo.
